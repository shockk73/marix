# Дизайн: передизайн видения системы LLM-моделью

**Дата:** 2026-06-12
**Автор:** brainstorming session (направление задал юзер, UX-решения — Claude)

## Проблема

Системный промпт — свалка из ~20 накопленных правил. У модели нет целостной
картины мира: она мешает понятия в кучу, выписывает юзеру техническую кашу
(id, ключи, интервалы, статусы), отвечает простынями, предлагает автобронь
там, где её нет, и задаёт цепочки лишних вопросов.

## Цель

Агент быстро и непринуждённо доводит юзера до результата: минимум слов,
минимум вопросов, ноль внутренностей. Юзер — не гик и не должен знать механику.

## UX-принципы (зафиксированы)

1. **Реальность первая.** Запрос «хочу ехать / есть места?» → сразу
   `check_trips_now`, ответ от реальной картины, а не анкета.
2. **Мест нет → слежка в одну кнопку.** Предлагается экраном
   («🔔 Следить» / «Не надо»). Если юзер сразу сказал «следи» — создаётся
   молча, подтверждение одной фразой.
3. **Бронь существует только там, где она есть.** Только `baranovichi_express`
   с подключённым аккаунтом. Другим перевозчикам бронь не упоминать.
   Аккаунт не подключён — предложить один раз.
4. **Дефолты вместо вопросов.** Интервал = 120с всегда (юзера НЕ спрашивать,
   параметр становится опциональным); окно не названо = весь день; остановки
   не названы = главные (озвучиваются в подтверждении брони).
   Недостающее спрашивается ОДНОЙ формой `ask_user_form`, не цепочкой.
5. **Стиль: 1–3 коротких предложения.** Без списков/заголовков без запроса,
   без техдеталей, механика — только на прямой вопрос (1–2 фразы),
   подтверждение действия — одна фраза + максимум один следующий шаг.

## Новая структура промпта (`build_system_prompt`, перезапись)

1. **Миссия** — 2 строки: кто такой, главная цель (результат минимумом
   слов и кликов).
2. **Память** — срез состояния (рендер сохраняется как есть), с явной
   пометкой «юзеру не показывать».
3. **Модель мира** — явные понятия и связи:
   Рейс (автобус: время/места/цена) → Поездка/ЦЕЛЬ (направление+дата+окно;
   все слежки одного вызова create_watch = одна поездка; «через несколько
   перевозчиков» = один вызов с несколькими providers; добавить перевозчика =
   goal_id) → Слежка (робот; успешная бронь закрывает все слежки поездки;
   истёкшие гаснут сами) → Бронь (только барановичи + аккаунт) →
   Приоритетное время (бронь вне него временная, перебронирование по кнопке,
   само не происходит) → Остановки (посадка/высадка; вслепую не бронируем).
4. **Главный флоу** — дерево решений из UX-принципов 1–4.
5. **Стиль** — жёсткие правила + примеры «плохо/хорошо» (создание слежки,
   приветствие).
6. **Справочник для tool calls** — компактно: провайдеры/направления,
   диагностика (execution, atlas proxy), формы/экраны, системные
   подтверждения, markdown-правила, кнопки клавиатуры, онбординг, оффтоп.

## Изменения в коде

- `llm/prompt.py` — перезапись по структуре выше; `LLM_SESSION_VERSION` bump.
- `llm/tools.py` — `create_watch.interval_sec` опционален (дефолт 120),
  описание запрещает спрашивать юзера.
- `llm/agent.py` — «🔧»-лейблы только для медленных операций
  (check_trips_now, бронь/отмена, логин, остановки, отчёт); быстрые тулзы
  выполняются молча; нет fallback-лейбла «Выполняю …».

## Проверка

- Обновлённые юнит-тесты промпта/тулзов/лейблов.
- **Живой прогон** через реальный OpenRouter (gemini-3.1-flash-lite),
  скриптом без Telegram: 5–6 сценариев («привет», «надо завтра утром в
  Барановичи», «следи», «что у меня», «как ты устроен», оффтоп) —
  критерии: ≤3 предложений, ноль техдеталей/ключей, правильные tool calls,
  бронь упоминается только к месту.

## Вне скоупа

- Изменение механик (цели, брони, подтверждения работают как есть).
- FSM `/watch` — нетронутый fallback.
