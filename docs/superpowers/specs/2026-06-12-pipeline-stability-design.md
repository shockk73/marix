# Дизайн: стабильность и отказоустойчивость пайплайна

**Дата:** 2026-06-12
**Автор:** brainstorming session
**Порядок:** под-проект 7 из 7 — сквозной, элементы можно делать вместе с остальными

## Цель

Ни одно появившееся место не теряется молча; ни одна ошибка не оставляет юзера
в тишине; сломавшаяся слежка сама сообщает о себе и не долбит сайт.

## Найденные проблемы (аудит 2026-06-12)

| # | Проблема | Где |
|---|---|---|
| 1 | `update_notified_trips` пишется ДО `_send_notification` — упавшая отправка теряет уведомление навсегда | `scheduler.py:106-108` |
| 2 | Исключение из `dispatch_tool` пробрасывается из `_drive_turn`: юзеру тишина, в истории висячий `tool_call` без результата → возможный 400 на следующем ходу | `llm/agent.py:379` |
| 3 | Слежка с вечными ошибками молчит и опрашивает с тем же интервалом (нет алерта, нет backoff) | `scheduler.py:_poll_loop` |
| 4 | Нет обработки флуд-контроля Telegram (`TelegramRetryAfter`) при отправках | scheduler + agent |
| 5 | SQLite без WAL: параллельные слежки + хендлеры могут ловить `database is locked` | `db.py` |

Что уже хорошо (не трогаем): LLM-ошибки доходят текстом, markdown-fallback,
`restart: unless-stopped`, восстановление watches/callbacks после рестарта,
ретраи клиента OpenRouter на 5xx/429, ретраи atlasbus на 429.

## Изменения

### 1. Доставка уведомлений: at-least-once

В `_poll_loop`: сначала `_send_notification`, потом `update_notified_trips`.
Упала отправка → исключение ловится существующим `except`, рейсы НЕ помечены,
следующий тик попробует снова. Дубль при падении между send и update —
приемлемая цена (лучше дважды, чем никогда).

### 2. Telegram-отправки с ретраем

Хелпер `send_with_retry(bot, chat_id, ...)`: ловит `TelegramRetryAfter`,
ждёт `retry_after` (cap 30с), повторяет один раз. Используется в
`_send_notification` и для алертов слежек. Сообщения агента не оборачиваем —
там есть свой fallback-путь и потеря реплики не критична.

### 3. Изоляция tool-вызовов

В `_drive_turn` обернуть `dispatch_tool` в try/except: исключение →
`insert_chat_message(role="tool", content='{"error": "Tool crashed: ..."}')`.
История остаётся консистентной (каждый tool_call получает результат),
модель видит ошибку и сама объясняет юзеру.

### 4. Здоровье слежек: алерт + backoff

В `_poll_loop` после `mark_watch_check_error`:
- `consecutive_errors == 10` (порог) → юзеру одно сообщение:
  «⚠️ Слежка #N спотыкается: <короткая ошибка>. Продолжаю пытаться.»
- Успех после ≥10 ошибок → «✅ Слежка #N снова работает.»
- Backoff: интервал сна = `interval_sec * min(2 ** (consecutive_errors // 5), 8)`
  (т.е. ×2 после 5 ошибок, ×4 после 10, cap ×8). Успех сбрасывает на базовый.
- `consecutive_errors` уже считается в `watch_statuses` — переиспользуем,
  в `_poll_loop` держим локальный счётчик (вместо чтения БД на каждом тике).

### 5. SQLite WAL

В `init_db`: `PRAGMA journal_mode=WAL` + `PRAGMA busy_timeout=5000`.

### 6. Смерть watch-таски

`asyncio.Task.add_done_callback` в `start_watch`: если таска завершилась
с не-CancelledError исключением — `logger.critical` + попытка уведомить юзера
«слежка #N упала, перезапусти бота или создай заново». (Практически
недостижимо — `_poll_loop` ловит всё, — но дешёвая страховка.)

## Тесты

- Отправка уведомления упала → notified_trips НЕ обновлён; следующий тик
  шлёт снова (мок бота: первый раз кидает, второй — ок).
- `TelegramRetryAfter` → подождали, повторили, доставлено.
- `dispatch_tool` кидает → в истории tool-result с error, юзер получает ответ
  модели, следующий ход не 400-ит.
- 10 подряд ошибок → один алерт (не десять); восстановление → одно сообщение.
- Backoff: интервалы растут по формуле, успех сбрасывает.

## Вне скоупа

- Метрики/мониторинг (Prometheus и т.п.), очередь уведомлений в БД,
  несколько инстансов бота, healthcheck-эндпоинт.
