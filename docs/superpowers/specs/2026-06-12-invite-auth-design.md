# Дизайн: invite-авторизация и роли

**Дата:** 2026-06-12
**Автор:** brainstorming session
**Порядок:** под-проект 3 из 6

## Цель

Новый юзер попадает в бота **одним кликом по ссылке-инвайту** — без команд и кодов.
Мастер-код из `.env` поднимает до админа; админ раздаёт инвайты.

## Решения, принятые на brainstorming

| Решение | Значение |
|---|---|
| Формат инвайта | Одноразовая deep-link `t.me/<бот>?start=<токен>`; один токен — один юзер |
| Подъём до админа | `AUTH_CODE` из `.env`, присланный текстом или через `/auth <код>` |
| Существующие юзеры | Остаются авторизованными с ролью `user` |
| Создание инвайтов | LLM-first: админ пишет «дай инвайт» → tool `create_invite` (см. спеку UX) |
| Срок жизни инвайта | Бессрочный, пока не использован (бот приватный, токен случайный) |

## Изменения в БД (`db.py`)

1. `authorized_users` + колонка `role TEXT NOT NULL DEFAULT 'user'`
   (значения `user` / `admin`). Миграция: `ALTER TABLE ... ADD COLUMN` в `init_db`
   с проверкой существования колонки (паттерн `PRAGMA table_info`).
2. Новая таблица:

```sql
CREATE TABLE IF NOT EXISTS invites (
    token      TEXT PRIMARY KEY,      -- secrets.token_urlsafe(12)
    created_by INTEGER NOT NULL,
    created_at TEXT NOT NULL,
    used_by    INTEGER,
    used_at    TEXT
)
```

3. Новые функции: `create_invite(created_by) -> token`,
   `use_invite(token, user_id) -> bool` (атомарно: UPDATE ... WHERE used_by IS NULL),
   `list_invites() -> list[dict]`, `get_user_role(user_id) -> str | None`,
   `set_user_role(user_id, role)`. `authorize_user(user_id, role="user")` расширяется
   параметром роли.

## Поток входа по инвайту

1. Юзер кликает `t.me/<бот>?start=<токен>` → Telegram шлёт `/start <токен>`.
2. `AuthMiddleware` пропускает `/start` с payload до проверки авторизации
   (сейчас middleware блокирует всё; нужно: если юзер не авторизован и
   сообщение — `/start <payload>`, попробовать `use_invite`).
3. Токен валиден → `authorize_user(user_id)`, токен помечается использованным,
   агент шлёт приветствие-онбординг (см. спеку UX).
4. Токен невалиден/использован → «Ссылка не работает или уже использована.
   Попроси новую у того, кто тебя пригласил.»
5. Попытки подбора токена НЕ инкрементят счётчик бана (юзер мог получить
   протухшую ссылку легально); бан остаётся только за подбор `AUTH_CODE`.

## Подъём до админа

- Неавторизованный или авторизованный юзер присылает `AUTH_CODE` —
  простым текстом (точное совпадение строки) или `/auth <код>`.
- Совпало → `authorize_user(user_id, role="admin")` / `set_user_role(..., "admin")`,
  ответ «Ты админ. Напиши "дай инвайт", чтобы пригласить человека.»
- Не совпало → существующая логика попыток и бана (3 попытки) без изменений.
- Plain-text проверка кода живёт в `AuthMiddleware` (для неавторизованных)
  и в текстовом fallback-хендлере ДО передачи в LLM (для авторизованных,
  чтобы код не утекал в историю LLM): если `message.text.strip() == AUTH_CODE` —
  обработать и не передавать агенту.

## LLM-инструменты (детали интеграции в спеке UX)

- `create_invite()` → `{token, link}`; только `role=admin`.
- `list_invites()` → список с статусами (ждёт / использован кем, когда); только админ.
- Фильтрация: `build_tools_for_role(role)` отдаёт админ-инструменты только админам,
  плюс проверка роли внутри хендлеров инструментов (защита в глубину).

## Тесты

- `use_invite`: успех, повторное использование → false, несуществующий токен → false.
- Миграция роли: старая запись `authorized_users` получает роль `user`.
- Middleware: неавторизованный + валидный инвайт → авторизован; невалидный → отказ
  без бана; `AUTH_CODE` текстом → роль admin.
- Гонка: два юзера используют один токен одновременно → авторизуется один
  (атомарный UPDATE).

## Вне скоупа

- Отзыв/срок жизни инвайтов, несколько админ-уровней, удаление юзеров.
