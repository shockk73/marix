# Дизайн: LLM-first UX

**Дата:** 2026-06-12
**Автор:** brainstorming session
**Порядок:** под-проект 4 из 6

## Цель

Юзер, который «не понимает в команды», пользуется ботом через обычный диалог.
LLM — главный интерфейс: сама ведёт сценарии и **сама собирает экраны**
(текст + сетки инлайн-кнопок), когда выбор удобнее клика.

## Решения, принятые на brainstorming

| Решение | Значение |
|---|---|
| Главный интерфейс | LLM-диалог (текст/голос/фото). Команды `/watch` `/list` `/stop` остаются нетронутым fallback'ом |
| Экраны | Новый tool `show_screen`: markdown-текст + сетка инлайн-кнопок, собирается моделью динамически |
| Reply-клавиатура | 3–4 постоянные кнопки-подсказки, каждая отправляет готовую фразу агенту (не FSM) |
| Онбординг | После входа по инвайту агент сам здоровается и показывает стартовый экран |
| Админ-функции | Те же диалоги: «дай инвайт», «покажи отчёт» → админ-инструменты |

## Tool `show_screen`

Обобщение существующего `ask_user` (столбик ≤8 кнопок) до экрана:

```json
{
  "name": "show_screen",
  "parameters": {
    "text": "string, markdown-текст экрана",
    "buttons": [[{"label": "🌅 Утро", "value": "утро 05:00-12:00"}]]
  }
}
```

- `buttons` — список рядов, ряд — список кнопок `{label, value}`; ≤ 8 рядов × 4 кнопки.
- Клик по кнопке → `value` возвращается агенту как выбор юзера (продолжение
  turn'а), ровно как сейчас работает `ask_user` через `pending_tool_calls`.
- Механика хранения: переиспользуем `pending_tool_calls`; `options_json` хранит
  плоский список values + раскладку рядов; `callback_data` остаётся `ai:<idx>`
  (плоский индекс). Одновременно активен один экран на юзера — клик по старому
  экрану отвечает «Вопрос устарел» (текущее поведение).
- `ask_user` остаётся для простых вопросов (модели проще), `show_screen` — для
  сеток: выбор даты (7 дней), пресеты времени, интервалы, подтверждения,
  карточки слежек с кнопками «⏹ Остановить #12» (value: «останови слежку 12»).

## Reply-клавиатура

- Кнопки: «🔍 Следить за местами», «📋 Мои слежки», «❓ Что ты умеешь».
  У админа четвёртая: «🛠 Админка». Кнопка «🎫 Подключить бронь» добавится
  в под-проекте автоброни (6).
- Нажатие = отправка этого текста в чат → попадает в обычный LLM-флоу.
  В системном промпте — маппинг фраз на ожидаемые сценарии.
- Показывается: после авторизации, по `/start`, после онбординга.
- `ReplyKeyboardMarkup(resize_keyboard=True, is_persistent=True)`.

## Онбординг

После успешного `use_invite` хендлер вызывает `agent.run_turn` с синтетическим
текстом-триггером (например `[новый пользователь вошёл по инвайту]`), промпт
велит агенту: поздороваться, в двух фразах объяснить что умеет, показать
`show_screen` со стартовыми действиями. Reply-клавиатура прикладывается
к приветствию.

## Изменения системного промпта (`llm/prompt.py`)

- Описать `show_screen`: когда строить экран (выбор из конечного набора:
  даты, времена, интервалы, подтверждения), когда НЕ строить (свободный ввод);
  всегда давать юзеру возможность ответить текстом вместо клика.
- Сценарий «создание слежки»: собрать провайдера(ов) → направление → дату
  (экран: ближайшие 7 дней) → окно времени (экран: 🌅 Утро 05–12 / 🌞 День 12–17 /
  🌆 Вечер 17–23 / Весь день / своё) → интервал (1/2/5/10 мин) → `create_watch`.
- Сценарий «мои слежки»: `list_watches` → карточки + экран с кнопками остановки.
- Маппинг кнопок reply-клавиатуры на сценарии.
- Поднять `LLM_SESSION_VERSION`.

## Изменения в коде

- `llm/tools.py`: схема `show_screen`; в `dispatch_tool` он, как и `ask_user`,
  обрабатывается агентом (рендер кнопок), не хендлером.
- `llm/agent.py`: `_handle_ask_user` обобщается до `_handle_screen`
  (рендер `InlineKeyboardMarkup` из рядов; сохранение pending).
- `handlers.py`: reply-клавиатура в `/start` и онбординге; админ-кнопка
  показывается по роли.
- `bot.py`: без изменений.

## Тесты

- `show_screen`: рендер сетки в InlineKeyboardMarkup, клик возвращает value
  агенту, второй клик по устаревшему экрану → «Вопрос устарел».
- Лимиты: >8 рядов или >4 кнопок в ряду → ошибка инструменту (агент переделает).
- Reply-клавиатура: текст кнопки попадает в LLM-флоу как обычное сообщение.
- Онбординг: `use_invite` → агент вызван с триггер-текстом.

## Вне скоупа

- Удаление FSM-мастера `/watch` (работает — не трогаем).
- Несколько одновременных экранов, web-app кнопки, пагинация.
