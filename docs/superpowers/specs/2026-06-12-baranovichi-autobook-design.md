# Дизайн: автобронирование Барановичи Экспресс

**Дата:** 2026-06-12
**Автор:** brainstorming session
**Порядок:** под-проект 6 из 6 (последний — самый рискованный)

## Цель

Когда на отслеживаемом рейсе появляется место, бот может сам его забронировать
под аккаунтом юзера на `tickets.baranovichi-express.by` и уведомить.

## Решения, принятые на brainstorming

| Решение | Значение |
|---|---|
| Чей аккаунт | Каждый юзер привязывает СВОЙ аккаунт сайта (бронь = телефон аккаунта, водитель сверяет при посадке) |
| Ввод кредов | Через LLM: юзер пишет телефон+пароль агенту, tool делает пробный логин и сохраняет |
| Режимы брони | Оба на выбор per-watch: «🔕 уведомить» (дефолт, как сейчас) / «🔔 спросить» (кнопка в уведомлении) / «⚡ авто» |
| Мест за бронь | Всегда 1 |
| После успешной брони | Слежка останавливается, юзер получает подтверждение |
| Хранение пароля | Открытым текстом в SQLite (как вся БД; бот приватный, сервер свой) + затирание пароля в истории чата |

## Разведка сайта (2026-06-12)

- Регистрация: телефон+пароль+имя, SMS-верификации нет. Логин: POST `/login`
  (`_token` CSRF со страницы + `partner_id` + `username` + `password`), сессия в куках.
- Бронь — резервация на номер телефона аккаунта, оплата водителю наличными.
  Онлайн-оплаты нет → «бронь» не списывает деньги.
- ⚠️ **Эндпоинт брони виден только под логином.** Кнопка «Купить» для анонима
  открывает модалку входа. Снять реальный booking-запрос можно только с живым
  аккаунтом — см. «Этап разведки» ниже.

## Этап разведки (первый шаг реализации, нужен юзер)

1. Юзер даёт креды своего аккаунта (или регистрирует тестовый — регистрация
   свободная, без SMS).
2. Логинимся скриптом, забираем HTML выдачи поиска под логином → видим форму/URL
   кнопки «Купить».
3. Проверяем `/user/tickets/active`: есть ли отмена брони. Если есть — безопасно
   делаем одну тестовую бронь и сразу отменяем. Если нет — тестовую бронь делаем
   только на рейс, на который юзер реально поедет, по согласованию.
4. Фиксируем запрос в спеке/коде (`book_trip`), дальше юзер не нужен.

## Изменения в БД

```sql
CREATE TABLE IF NOT EXISTS site_credentials (
    user_id    INTEGER PRIMARY KEY,   -- TG user
    site       TEXT NOT NULL DEFAULT 'baranovichi_express',
    phone      TEXT NOT NULL,
    password   TEXT NOT NULL,
    verified_at TEXT,
    created_at TEXT NOT NULL
)
```

`watches` + колонка `autobook TEXT NOT NULL DEFAULT 'off'` (`off`/`confirm`/`auto`),
миграция через `PRAGMA table_info`.

## Клиент сайта: `providers/baranovichi_session.py`

- `login(client, phone, password) -> cookies` — GET `/login` (берём `_token`),
  POST `/login`, проверка успеха (редирект/наличие «Выйти»). Ошибка кредов →
  типизированное исключение `InvalidCredentials`.
- `book_trip(client, cookies, trip, seats=1) -> BookingResult` — запрос со
  страницы выдачи под логином (заполняется на этапе разведки). Результат:
  успех / место уже занято / сессия протухла (→ один перелогин и повтор).
- Куки кэшируются в памяти per-user (dict), при протухании — перелогин.

## LLM-инструменты

- `save_baranovichi_credentials(phone, password)`: пробный `login()`; успех →
  сохранить, ответ «Подключено». Провал → «Логин/пароль не подходят».
  **После обработки пароль затирается в `chat_messages`** (UPDATE content
  заменой пароля на `***`) — не светится в контексте и админ-отчёте.
- `get_credentials_status()` → подключено/нет, телефон маской (`+375…67`).
- `delete_credentials()` → удалить запись.
- `create_watch` получает опциональный параметр `autobook` (`off|confirm|auto`):
  валиден только для `baranovichi_express`; `confirm`/`auto` требуют сохранённых
  кредов (иначе ошибка инструменту — агент предложит подключить).

## Поток в `scheduler.py`

В `_poll_loop`, после `compute_newly_available`, если watch принадлежит
`baranovichi_express` и `autobook != off` и есть новые места:

- `auto`: берём самый ранний новый рейс → `book_trip` → успех: уведомление
  «✅ Забронировал рейс 07:00, Минск → Барановичи, 13.06. Водитель сверит твой
  номер при посадке», слежка останавливается (`stop_watch` + cancel). Провал
  «место занято»: продолжаем следить, уведомление не шлём (место уже ушло).
  Ошибка логина: уведомить «креды не работают, бронь выключена» и переключить
  watch в `off`.
- `confirm`: обычное уведомление + инлайн-кнопка «🎫 Забронировать 07:00»
  (callback `book:<watch_id>:<trip_id>`). Клик → бронь тем же путём; место
  ушло → «Не успели, место уже заняли — продолжаю следить».
- Бронируем только рейсы из новых (`newly`), чтобы не бронировать повторно.

## Тесты

- `login`: успех (мок-страницы), неверные креды, протухшая сессия → перелогин.
- `book_trip`: успех / занято / протухло (respx-моки по снятому запросу).
- Scheduler `auto`: новое место → бронь → уведомление → слежка остановлена;
  бронь не удалась (занято) → слежка живёт.
- `confirm`: кнопка появляется; клик → бронь; повторный клик → идемпотентность.
- `create_watch` с `autobook` без кредов → ошибка.
- Затирание пароля в истории после `save_baranovichi_credentials`.

## Риски

- Сайт может менять вёрстку/анти-бот: ловим, алертим юзеру, бронь деградирует
  в «🔕 уведомить».
- Двойная бронь при рестарте между book и stop_watch: после успешной брони
  сначала `stop_watch` в БД, потом уведомление.

## Вне скоупа

- Шифрование паролей, бронь >1 места, отмена брони из бота, другие сайты.
